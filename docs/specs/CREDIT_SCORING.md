## Description

Build an explainable, regulator-ready alternative credit scoring system for Vietnam SMEs and household merchants using normalized transaction data. The production decision path must remain fully auditable, deterministic, and interpretable, with WoE logistic scorecards as the only allowed statistical scoring model and fixed reason codes as the only allowed explanation vocabulary.

## User Stories

- As a **credit risk engine**, I want to score merchants from multi-source transaction data, so that thin-file SMEs can receive prudent credit offers.
- As a **compliance reviewer**, I want every score, feature, rule, limit, and explanation to be reproducible from audit logs, so that SBV prudence and documentation requirements are satisfied.
- As a **merchant applicant**, I want fixed, understandable positive and negative factors with concrete improvement guidance, so that lending decisions are explainable and actionable.
- As an **engineer**, I want strict schemas, rule thresholds, score bands, and output contracts, so that the system can be implemented without ambiguous behavior.

## Normative Product Decisions

1. All upstream data must be normalized into one canonical ledger before feature generation.
2. The only production scoring model is a **WoE logistic scorecard**.
3. **XGBoost, neural nets, LLM scoring, and any black-box decision models are forbidden** in the production decision path.
4. All reasons must come from a **fixed catalog of 27 standardized reason codes**.
5. LLMs may only translate pre-approved reason codes into approved surface text; they may not generate, rank, or modify reasons.
6. Fraud detection must be **rule-based and auditable**.
7. Thin-file merchants (`<30` days history) must use a reduced-feature fallback path with lower score/limit ceilings.
8. Phone-only entity resolution may continue to scoring, but the final score must be capped at `650` and the entity may not receive limits above the starter schedule until stronger verification is added.
9. Household-business data must be treated as personal data under PDPD; owner-level consent is required before scoring.

## Domain Constants and Enumerations

### 1) Canonical Ledger Schema (14 fields)

Every normalized transaction record must contain exactly these 14 fields.

| # | Field | Type | Required | Rules |
|---|-------|------|----------|-------|
| 1 | `ledger_txn_id` | UUID/string | Yes | Immutable ID generated by ingestion layer. |
| 2 | `canonical_entity_id` | string | Yes | Resolved merchant/business entity ID used downstream. |
| 3 | `source_type` | enum | Yes | `BANK`, `EWALLET`, `MARKETPLACE`, `POS`, `PAYROLL`, `INVOICE`, `UTILITY`, `SELF_REPORTED`, `OTHER`. |
| 4 | `source_record_id` | string | Yes | Native provider transaction identifier; must be unique within `source_type`. |
| 5 | `event_timestamp` | ISO-8601 datetime | Yes | Original business event timestamp in UTC; if source is local time only, convert and persist original timezone in raw audit payload. |
| 6 | `settlement_timestamp` | ISO-8601 datetime/null | No | When funds settled; nullable if source has no settlement concept. |
| 7 | `currency_code` | string | Yes | ISO-4217; production default is `VND`; all model features use post-FX normalized VND amount. |
| 8 | `amount_minor` | integer | Yes | Positive minor units only; sign is derived from `direction`. |
| 9 | `direction` | enum | Yes | `INFLOW` or `OUTFLOW`. |
| 10 | `counterparty_ref` | string | Yes | Normalized payer/payee identifier or salted hash when PII must be masked. |
| 11 | `channel` | enum | Yes | `BANK_TRANSFER`, `EWALLET`, `CARD_POS`, `SOFTPOS`, `MARKETPLACE`, `CASH_DEPOSIT`, `INVOICE`, `BILL_PAYMENT`, `SELF_REPORTED`, `OTHER`. |
| 12 | `transaction_type` | enum | Yes | `SALE`, `REFUND`, `PAYOUT`, `TRANSFER`, `PAYMENT`, `FEE`, `CHARGEBACK`, `ADJUSTMENT`, `OTHER`. |
| 13 | `confidence_score` | decimal(4,3) | Yes | Value in `[0.000,1.000]`; derived from source weight, data completeness, authenticity, and identity confidence. |
| 14 | `raw_record_hash` | string | Yes | SHA-256 hash of canonicalized raw payload for immutability and replay. |

#### Canonical Ledger Acceptance Criteria

- Records missing any required field must be rejected from scoring and written to an ingestion error log.
- `amount_minor` must never be negative.
- `direction=INFLOW` and `direction=OUTFLOW` must be the only allowed direction values.
- `confidence_score` must be rounded to 3 decimals and clamped to `[0,1]`.
- Duplicate `source_type + source_record_id` combinations must not create duplicate ledger rows.
- All derived features must reference `ledger_txn_id` lineage in audit logs.

### 2) Source Type Confidence Weighting

The baseline source trust ordering is mandatory: **bank > e-wallet > POS > self-reported**.

| `source_type` | Baseline weight |
|---------------|-----------------|
| `BANK` | `1.00` |
| `EWALLET` | `0.90` |
| `MARKETPLACE` | `0.85` |
| `POS` | `0.80` |
| `PAYROLL` | `0.78` |
| `INVOICE` | `0.74` |
| `UTILITY` | `0.70` |
| `SELF_REPORTED` | `0.40` |
| `OTHER` | `0.50` |

`confidence_score` must be computed as:

`baseline_source_weight x connector_authenticity_multiplier x field_completeness_multiplier x entity_resolution_multiplier x duplicate_penalty_multiplier`

Required multipliers:

- `connector_authenticity_multiplier`: `1.00` API/bank statement import, `0.85` signed CSV import, `0.60` manual upload, `0.40` self-entry.
- `field_completeness_multiplier`: `1.00` all required fields present, `0.90` only optional field missing, otherwise reject.
- `entity_resolution_multiplier`: tier-based from entity-resolution section below.
- `duplicate_penalty_multiplier`: `1.00` no duplication, `0.50` suspected duplicate, `0.00` confirmed duplicate row rejected.

### 3) Entity Resolution Rules

#### Matching Hierarchy

1. `BUSINESS_ID` exact match
2. `BANK_ACCOUNT` exact match
3. `PHONE + NAME + ADDRESS_HASH` composite match
4. `PHONE_ONLY` fallback

| Resolution tier | Match rule | Resolution confidence | `entity_resolution_multiplier` | Score cap | Limit gate |
|-----------------|-----------|-----------------------|-------------------------------|----------|------------|
| `TIER_1_BUSINESS_ID` | Exact legal business registration ID | `1.00` | `1.00` | None | Full |
| `TIER_2_BANK_ACCOUNT` | Exact bank account ownership match | `0.95` | `0.95` | None | Full |
| `TIER_3_COMPOSITE` | Phone + normalized name + address hash | `0.80` | `0.85` | None | Full with duplicate checks |
| `TIER_4_PHONE_ONLY` | Phone only | `0.55` | `0.65` | `650` | Starter ceiling only |

#### Duplicate Detection Rules

- If one raw identity maps to multiple `canonical_entity_id` values, flag `DUPLICATE_IDENTITY_CONFLICT` and block automated limit issuance.
- If multiple source connectors yield materially identical transactions (`same amount`, `same day`, `same counterparty_ref`, `same type`) within 5 minutes, keep one canonical row and mark the rest duplicates.
- If a `PHONE_ONLY` entity later matches a `BUSINESS_ID` or `BANK_ACCOUNT`, the entity graph must merge under one `canonical_entity_id` and recompute historical features.

### 4) Score Bands

| Band | Score range | Meaning | Default lending action |
|------|-------------|---------|------------------------|
| `DECLINE` | `300-499` | High risk / insufficient prudence | No automated offer |
| `OBSERVE` | `500-579` | Weak but monitorable | Observation or micro-credit only |
| `STARTER` | `580-649` | Thin or moderate quality | Reduced limit schedule |
| `STANDARD` | `650-749` | Acceptable risk | Standard limit schedule |
| `PRIME` | `750-850` | Strong risk profile | Prime limit schedule |

### 5) Progressive Verification Gates

| Gate | Minimum evidence | Maximum first-loan exposure |
|------|------------------|-----------------------------|
| `G0_NO_VERIFIED_SOURCE` | Self-reported only or missing consent | `0` |
| `G1_SINGLE_SOURCE_PHONE_ONLY` | Phone-only entity + 1 verified transaction source | `USD 50 / VND 1,250,000` |
| `G2_SINGLE_SOURCE_VERIFIED` | Business/account/composite entity + 1 verified source | `USD 100 / VND 2,500,000` |
| `G3_MULTI_SOURCE_ENTRY` | 2 verified sources or 1 bank source + valid consent | Use band/history entrance schedule |
| `G4_MATURE` | 2+ verified sources + successful repayment history | Use mature ceiling schedule |

## Model Specification

### 1) Primary Model: WoE Logistic Scorecard

The production scoring path must use a logistic scorecard with binned features and Weight of Evidence.

#### Full-file model eligibility

Use the full-file model only when all are true:

- `history_length_days >= 30`
- `weighted_inflow_txn_count_30d >= 20`
- average source-weighted `confidence_score >= 0.65`
- valid PDPD consent exists when the merchant is a household business / sole proprietor

#### Full-file features

| Feature ID | Feature name | Definition | Window | Notes |
|------------|--------------|------------|--------|-------|
| `F01` | `history_length_days` | Days between first and latest confident transaction | lifetime | Used for stability and thin-file separation |
| `F02` | `entity_resolution_confidence` | Numeric confidence from entity tier | current | Used directly in scorecard and for caps |
| `F03` | `seasonality_adjusted_revenue_zscore` | Peer-cohort z-score of confidence-weighted inflows after seasonal normalization | 90d | Sector+province+month cohort |
| `F04` | `inflow_stability_cv` | Coefficient of variation of weekly net inflows | 90d | Lower is better |
| `F05` | `yoy_same_lunar_window_growth` | Growth vs same lunar-calendar window prior year | aligned window | Fallback rules below |
| `F06` | `fast_refund_rate` | Refund value posted within 72h / gross sales value | 30d | Higher is worse |
| `F07` | `inactive_gap_days` | Longest transaction gap | 90d | Lower is better |
| `F08` | `counterparty_diversity_ratio` | Unique paying counterparties / incoming transaction count, adjusted for circularity | 90d | Higher is better |
| `F09` | `cross_channel_variance` | Relative variance between channel-level sales totals | 90d | Lower is better |

#### Thin-file model eligibility

Use the thin-file model when either is true:

- `history_length_days < 30`, or
- `weighted_inflow_txn_count_30d < 20`

#### Thin-file feature set

| Feature ID | Feature name | Definition | Window |
|------------|--------------|------------|--------|
| `TF01` | `history_length_days` | Same as full-file | lifetime |
| `TF02` | `weighted_avg_daily_inflow` | Confidence-weighted daily inflow average normalized to sector benchmark | 14d |
| `TF03` | `active_days_ratio` | Active transaction days / calendar days | 14d |
| `TF04` | `counterparty_diversity_ratio` | Same metric with shorter lookback | 14d |
| `TF05` | `fast_refund_rate` | Same metric with shorter lookback | 14d |
| `TF06` | `entity_resolution_confidence` | Same as full-file | current |

#### Binning and WoE requirements

- Each feature must have explicit, versioned bin boundaries.
- Bins must be monotonic in expected risk direction.
- Full-file bins must target at least `5%` training support per bin unless an explicit exception is documented.
- Missing data must use a dedicated `MISSING` bin; median imputation is forbidden.
- WoE must be computed with additive smoothing of `0.5` good and bad counts per bin.
- Each bin must have a reason-code mapping defined before deployment.

#### Score formula

`logit = beta_0 + Σ(beta_i x WOE(bin_i))`

`factor = 20 / ln(2)`

`offset = 700 - factor x ln(20)`

`raw_score = offset + factor x logit`

`score_after_fraud = raw_score + fraud_penalty_points`

`final_score = clamp(300, min(score_after_fraud, resolution_score_cap), 850)`

Where:

- `fraud_penalty_points` is between `0` and `-150`
- `resolution_score_cap` is `650` only for `TIER_4_PHONE_ONLY`, otherwise `850`

### 2) Standardized 27 Reason Codes

The catalog below is fixed. No dynamic generation is allowed.

| Code | Feature family | Polarity | Canonical meaning | Improvement guidance key |
|------|----------------|----------|-------------------|--------------------------|
| `RC01` | History length | Positive | Strong operating history observed | `MAINTAIN_LONGER_TRACK_RECORD` |
| `RC02` | History length | Negative | Limited operating history observed | `BUILD_30_TO_90_DAYS_HISTORY` |
| `RC03` | History length | Neutral | Insufficient operating history to score fully | `COMPLETE_OBSERVATION_PERIOD` |
| `RC04` | Entity confidence | Positive | Business identity strongly verified | `KEEP_VERIFIED_BUSINESS_DETAILS_CURRENT` |
| `RC05` | Entity confidence | Negative | Identity match confidence is low | `ADD_STRONGER_IDENTITY_EVIDENCE` |
| `RC06` | Entity confidence | Neutral | Phone-only identity cap applied | `LINK_BANK_OR_BUSINESS_REGISTRATION` |
| `RC07` | Revenue strength | Positive | Seasonality-adjusted revenue is strong for peer group | `MAINTAIN_REVENUE_LEVELS` |
| `RC08` | Revenue strength | Negative | Seasonality-adjusted revenue is weak for peer group | `INCREASE_STABLE_REVENUE` |
| `RC09` | Revenue strength | Neutral | Revenue evidence is incomplete or low-confidence | `CONNECT_HIGHER_TRUST_DATA_SOURCES` |
| `RC10` | Cashflow stability | Positive | Cash inflows are stable over time | `MAINTAIN_STABLE_CASHFLOW` |
| `RC11` | Cashflow stability | Negative | Cash inflows are volatile | `REDUCE_CASHFLOW_VOLATILITY` |
| `RC12` | Cashflow stability | Neutral | Cashflow pattern is irregular or sparse | `INCREASE_CONSISTENT_TRANSACTION_ACTIVITY` |
| `RC13` | Seasonal growth | Positive | Revenue grew versus the same lunar period last year | `SUSTAIN_YEAR_OVER_YEAR_GROWTH` |
| `RC14` | Seasonal growth | Negative | Revenue declined versus the same lunar period last year | `IMPROVE_SEASONAL_RECOVERY` |
| `RC15` | Seasonal growth | Neutral | No same-lunar comparison window available | `BUILD_LONGER_OPERATING_HISTORY` |
| `RC16` | Refund behavior | Positive | Refund and reversal activity is low | `MAINTAIN_LOW_REFUND_RATE` |
| `RC17` | Refund behavior | Negative | Fast refund activity is elevated | `REDUCE_72_HOUR_REFUNDS` |
| `RC18` | Refund behavior | Neutral | Refund history is insufficient for stable assessment | `BUILD_MORE_VERIFIED_SALES_HISTORY` |
| `RC19` | Activity continuity | Positive | Business activity is continuous | `MAINTAIN_REGULAR_ACTIVITY` |
| `RC20` | Activity continuity | Negative | Extended inactive periods detected | `REDUCE_LONG_INACTIVE_GAPS` |
| `RC21` | Activity continuity | Neutral | Activity history is too sparse | `INCREASE_ACTIVE_SALES_DAYS` |
| `RC22` | Counterparty diversity | Positive | Customer base is diverse | `MAINTAIN_DIVERSE_CUSTOMERS` |
| `RC23` | Counterparty diversity | Negative | Counterparty base is concentrated or circular | `DIVERSIFY_CUSTOMER_BASE` |
| `RC24` | Counterparty diversity | Neutral | Counterparty coverage is insufficient | `INCREASE_VERIFIED_CUSTOMER_ACTIVITY` |
| `RC25` | Cross-channel consistency | Positive | Sales are consistent across channels | `MAINTAIN_CROSS_CHANNEL_CONSISTENCY` |
| `RC26` | Cross-channel consistency | Negative | Material inconsistency exists across channels | `ALIGN_REPORTED_AND_VERIFIED_CHANNELS` |
| `RC27` | Cross-channel consistency | Neutral | Single-channel evidence only | `CONNECT_ADDITIONAL_SALES_CHANNELS` |

#### Reason-code output requirements

- Every score response must return exactly `3` positive factors and exactly `3` negative factors.
- Factors must be ordered by absolute point contribution.
- If the reduced feature set yields fewer than 3 positive or negative contributors, the system must pad with the next highest same-sign contributors from scored features; dynamic reason generation remains forbidden.
- Neutral reason codes may appear only in `supplemental_codes`, not in the required top-3 positive/top-3 negative arrays.
- LLM translation may only receive `reason_code`, `locale`, `approved guidance key`, and canonical factor labels; it may not receive `score`, `PD`, raw coefficients, or unrestricted text.

### 3) Adverse Action Format

If the applicant is declined, score-capped, or offered a reduced limit because of risk/fraud/entity-confidence rules, the response must include an `adverse_action` object with this shape:

```json
{
  "required": true,
  "threshold": {
    "type": "score_or_rule_threshold",
    "value": "650",
    "comparison": "actual_below_threshold"
  },
  "factors": [
    {
      "reason_code": "RC17",
      "feature_name": "fast_refund_rate",
      "actual_value": "12.4%",
      "required_value": "<= 3.0%",
      "guidance_key": "REDUCE_72_HOUR_REFUNDS"
    }
  ],
  "improvement_guidance": [
    "REDUCE_72_HOUR_REFUNDS",
    "CONNECT_HIGHER_TRUST_DATA_SOURCES"
  ]
}
```

Mandatory rules:

- Include all negative factors contributing more than `10%` of total negative point impact.
- Include exact factor values and exact required thresholds.
- Do not include any factor not present in the canonical score output.
- If the only negative outcome is a phone-only cap, the adverse action must explicitly state that stronger identity evidence can lift the cap.

## Decisioning and Limits

### 1) Progressive Limit Schedule

#### First-loan entrance schedule

All first loans must obey a hard cap of **`USD 500` or `VND 12,500,000` equivalent**, whichever is lower.

| Score band | 7-29 days history | 30-89 days history | 90+ days history |
|------------|-------------------|--------------------|------------------|
| `DECLINE` | `0` | `0` | `0` |
| `OBSERVE` | `USD 50 / VND 1,250,000` | `USD 50 / VND 1,250,000` | `USD 100 / VND 2,500,000` |
| `STARTER` | `USD 100 / VND 2,500,000` | `USD 200 / VND 5,000,000` | `USD 300 / VND 7,500,000` |
| `STANDARD` | `USD 100 / VND 2,500,000` | `USD 350 / VND 8,750,000` | `USD 500 / VND 12,500,000` |
| `PRIME` | `USD 100 / VND 2,500,000` | `USD 500 / VND 12,500,000` | `USD 500 / VND 12,500,000` |

Additional constraints:

- `TIER_4_PHONE_ONLY` entities may not exceed the `STARTER` schedule even if the numeric score reaches `650`.
- `G0_NO_VERIFIED_SOURCE` must never receive an offer.
- Thin-file merchants (`<30` days) may only receive the `7-29 days` column.

#### Automatic step-up triggers

| Trigger step | Conditions | Increase rule |
|--------------|------------|---------------|
| `STEP_1` | 1 successful repayment, no payment delinquency, no severe fraud flags since origination | Increase next offer by `25%`, capped by band/history ceiling |
| `STEP_2` | 2 consecutive successful repayments, active verified data feed continues | Increase next offer by `50%` over entrance amount, capped by mature ceiling |
| `STEP_3` | 3 consecutive successful repayments, `history_length_days >= 180`, `G4_MATURE` evidence | Increase to mature ceiling |

#### Mature ceilings

| Score band | Mature ceiling |
|------------|----------------|
| `OBSERVE` | `USD 150 / VND 3,750,000` |
| `STARTER` | `USD 500 / VND 12,500,000` |
| `STANDARD` | `USD 1,000 / VND 25,000,000` |
| `PRIME` | `USD 2,000 / VND 50,000,000` |

### 2) Cold-start Mitigations

- New merchants with `<7` days of history receive an **observation-only benchmark score** and no loan offer.
- For days `7-29`, the thin-file model must initialize from an **industry benchmark default score** based on `sector + province`; if unavailable, use a global default of `560`.
- The first 30 days are an observation period even if a micro-credit offer is permitted.
- Micro-credit entrance for cold start must remain in the `USD 50-100 / VND 1,250,000-2,500,000` range.
- Progressive data sharing must unlock higher limits: more verified sources and stronger identity evidence move the merchant from `G1`/`G2` to `G3`/`G4`.

## Seasonality Normalization

### Mandatory Vietnam-specific factors

- `TET_LUNAR_WINDOW`: lunar January-February window
- `SUMMER_PEAK_WINDOW`: June-August
- `BACK_TO_SCHOOL_WINDOW`: August-September

### Required normalization behavior

1. All revenue-like and activity-like features must be computed on confidence-weighted ledger data.
2. All seasonal comparisons must align to the **same lunar window** for Tết-related periods.
3. All revenue-like features must include a peer normalization by `sector + province + month`.
4. If a cohort has fewer than `30` merchants, widen fallback in order to:
   - `sector + region + month`
   - `sector + national + month`
5. If no prior-year same-lunar window exists, set the growth feature to the `MISSING` bin and emit `RC15`.

### Derived feature requirements

- `yoy_same_lunar_window_growth`
- `summer_peak_ratio`
- `back_to_school_ratio`
- `peer_adjusted_zscore_by_sector_province_month`
- `post_tet_recovery_days`

## Fraud Detection Rules

Fraud controls are separate from the scorecard and must be rule-based, deterministic, and logged.

| Rule ID | Rule | Warning threshold | Severe threshold | Penalty |
|---------|------|-------------------|------------------|---------|
| `FR01` | Volume velocity ratio vs trailing baseline | `> 3.0x` trailing 8-week median | `> 5.0x` trailing 8-week median | `-20 / -50` |
| `FR02` | Refund timing (`<72h`) | `> 5%` of gross sales or `>=2` fast refunds | `> 10%` of gross sales or `>=3` fast refunds | `-30 / -60` |
| `FR03` | Inactive period detection | max gap `>7` days in 30d or `>14` days in 90d | max gap `>21` days or active-days ratio `<0.30` | `-15 / -40` |
| `FR04` | Counterparty diversity / circularity | unique payer ratio `<0.20` or top 3 payers `>70%` of inflows | circular in/out counterparties `>30%` of incoming value | `-20 / -50` |
| `FR05` | Cross-channel consistency | monthly channel variance `>20%` | monthly channel variance `>35%` or self-report delta `>50%` | `-20 / -50` |

Fraud decision rules:

- Aggregate fraud penalties may not exceed `-150` points.
- Two severe fraud flags in one scoring run require manual review or automated decline according to policy configuration.
- A confirmed duplicate-entity conflict plus any severe fraud flag must force `DECLINE`.
- Fraud rules must emit raw measured values into the audit log and explanation payload.

## LLM Safety Architecture

### Hard boundary

- LLM is allowed only for translation/rendering of pre-approved reason codes.
- LLM is forbidden from generating reasons, changing factor ordering, inferring approval status, or seeing the raw score.

### Two-stage render

1. **Canonical form**: scoring engine emits structured explanation JSON from fixed codes.
2. **Surface form**: renderer or LLM translates canonical code payload into the requested locale.

### Canonical explanation payload

```json
{
  "decision_id": "uuid",
  "locale": "vi-VN",
  "positive_factors": [
    {"reason_code": "RC07", "factor_label": "seasonality_adjusted_revenue_zscore", "guidance_key": "MAINTAIN_REVENUE_LEVELS"}
  ],
  "negative_factors": [
    {"reason_code": "RC17", "factor_label": "fast_refund_rate", "guidance_key": "REDUCE_72_HOUR_REFUNDS"}
  ],
  "supplemental_codes": ["RC06", "RC27"]
}
```

### LLM rendering requirements

- Input must be structured JSON only.
- Output must be grammar-bound and schema-validated before release.
- If validation fails, the system must fall back to deterministic template rendering.
- 100% of LLM interactions must be audit logged with request hash, response hash, model ID, timestamp, validator result, and operator/environment identifier.
- Raw score, PD, model coefficients, and raw feature vectors must never be sent to the LLM.

## Regulatory Compliance Requirements

### SBV / prudence

- The system must preserve sufficient documentation to demonstrate prudence and explainability for each score and lending decision.
- Every production model version must have documented feature definitions, bin boundaries, coefficients, and validation results.
- No non-deterministic operation may exist in the critical scoring path.

### PDPD

- Household-business data must be treated as personal data.
- Owner-level consent is mandatory before ingesting or scoring household-business transaction data.
- If consent is missing, expired, or revoked, automated scoring must stop and the application must be marked `CONSENT_REQUIRED`.

### Retention

| Record class | Minimum retention |
|--------------|-------------------|
| Accounting records | `10 years` |
| Operational logs | `5 years` |
| Individual score audit trails | `7 years` |
| System design / architecture docs | `20 years` |

Additional rule: retention periods run from the underlying transaction/event date, not from explanation-render time.

## Scenarios

### Scenario 1: Normalize verified transaction feeds into the canonical 14-field ledger

- **Given** bank, e-wallet, marketplace, POS, or self-reported transaction feeds are received
- **When** ingestion normalizes them for scoring
- **Then** each accepted transaction must be written using the 14-field canonical ledger schema with a deterministic `confidence_score`

Acceptance criteria:

- Unsupported enum values must fail validation.
- Missing required fields must reject the row from scoring.
- All accepted rows must retain a raw payload hash and source lineage.

### Scenario 2: Reject malformed or duplicated transactions before feature generation

- **Given** a connector submits duplicated or incomplete records
- **When** the canonicalization pipeline processes the batch
- **Then** duplicates must be deduplicated or rejected and malformed rows must appear in the ingestion error log

Acceptance criteria:

- Confirmed duplicates must not alter feature aggregates.
- The audit trail must record the reason for every rejection.

### Scenario 3: Resolve merchant identity using the required hierarchy

- **Given** multiple identity clues exist for a merchant
- **When** the entity resolver runs
- **Then** it must apply the hierarchy `Business ID > Bank account > Phone+Name+Address > Phone only` and persist the chosen tier and confidence

Acceptance criteria:

- Exact business ID match always outranks lower tiers.
- Phone-only scoring must apply the `650` score cap and starter-only limit gate.
- Entity merges must recompute historical features against the merged `canonical_entity_id`.

### Scenario 4: Score a full-file merchant with the WoE logistic scorecard

- **Given** a merchant has at least 30 days of eligible history and sufficient confidence-weighted inflow activity
- **When** the scoring engine calculates the score
- **Then** it must use the full-file WoE logistic model, documented bins, and the approved score formula

Acceptance criteria:

- Only binned WoE features may be passed to the logistic equation.
- Every used bin must be audit logged with feature name, bin label, WoE value, coefficient, and point contribution.
- The final score must be clamped to `300-850`.

### Scenario 5: Route thin-file merchants to the reduced-feature fallback path

- **Given** a merchant has fewer than 30 days of history or insufficient transaction count
- **When** the scoring engine evaluates the application
- **Then** it must use the thin-file model and cold-start rules instead of the full-file model

Acceptance criteria:

- New merchants with fewer than 7 days of history must receive observation only and no loan offer.
- Thin-file offers must obey the reduced entrance schedule and micro-credit range.
- Thin-file explanations must still return exactly 3 positive and 3 negative factors.

### Scenario 6: Produce only fixed reason codes and adverse-action data

- **Given** a score and lending decision have been computed
- **When** the explanation layer prepares the response
- **Then** it must select from the 27-code catalog and include the top 3 positive and top 3 negative factors

Acceptance criteria:

- Dynamic reason generation is forbidden.
- Negative decisions must include the adverse-action payload with threshold, actual values, required values, and improvement guidance.
- Codes contributing more than 10% of negative impact must appear in adverse action.

### Scenario 7: Normalize seasonal business behavior for Vietnam-specific cycles

- **Given** a merchant operates through Tết, summer peak, or back-to-school cycles
- **When** revenue and stability features are computed
- **Then** the engine must compare the merchant against the same lunar window and the peer cohort `sector + province + month`

Acceptance criteria:

- Tết must use a lunar-aligned window.
- Cohorts with fewer than 30 merchants must use the documented fallback hierarchy.
- Missing prior-year comparison must map to the `MISSING` bin and `RC15`.

### Scenario 8: Apply auditable rule-based fraud penalties

- **Given** the merchant exhibits anomalous volume, refund, inactivity, counterparty, or channel behavior
- **When** fraud rules evaluate the application
- **Then** the engine must apply only the documented rule thresholds and penalties

Acceptance criteria:

- Penalties must be numeric and reproducible.
- The audit log must store measured values, threshold crossed, severity, and points deducted.
- Aggregate penalties may not exceed `-150`.

### Scenario 9: Issue a limit offer using score band, history length, and verification gate

- **Given** a final score exists and the merchant passed consent and fraud gates
- **When** underwriting sets the credit limit
- **Then** the offer must use the progressive limit schedule and verification-gate ceilings

Acceptance criteria:

- First-loan offers must never exceed `USD 500 / VND 12,500,000`.
- `G0_NO_VERIFIED_SOURCE` must never receive a limit.
- Phone-only entities must not exceed starter schedule.

### Scenario 10: Increase limits automatically after successful repayments

- **Given** a merchant has one or more successful repayments and continued verified data sharing
- **When** the next underwriting cycle runs
- **Then** the offer must step up according to the documented trigger table

Acceptance criteria:

- Delinquency or severe fraud flags must block automatic step-up.
- Mature ceilings require `G4_MATURE` evidence and 180+ days history.

### Scenario 11: Render explanations with a hard LLM safety boundary

- **Given** a localized explanation is requested
- **When** the rendering layer invokes deterministic templates or an LLM translator
- **Then** the model must only receive structured reason-code payloads and all interactions must be logged

Acceptance criteria:

- Raw score and coefficients must never be present in the LLM payload.
- Output must validate against the approved schema or fall back to deterministic templates.
- 100% of prompt/response pairs must be audit logged.

### Scenario 12: Enforce consent, documentation, and retention requirements

- **Given** a merchant is a household business or personal-data subject under PDPD
- **When** the platform ingests data, scores the merchant, and stores artifacts
- **Then** it must require owner-level consent and retain records according to the mandated retention schedule

Acceptance criteria:

- Missing or revoked consent must block automated scoring.
- Score trails, operational logs, accounting records, and system design artifacts must follow their minimum retention periods.
- Each score must be reproducible from its `audit_log_id`.

## Validation Rules

- The production decision path must not call any black-box model.
- All production score outputs must be reproducible from canonical ledger rows plus the versioned model package.
- Exactly 27 reason codes must exist; additions require a formal spec revision.
- Every score response must return exactly 3 positive and exactly 3 negative factors.
- Every adverse action must include exact factor values and exact required thresholds.
- Every ledger record used for scoring must have `confidence_score >= 0.250`; lower-confidence records may be stored but must be excluded from model aggregates.
- Observation-only cases (`history < 7 days`) must not produce a credit offer.
- Thin-file cases must never bypass the reduced-feature model.
- Fraud penalties must only use the five documented rule families.
- Phone-only resolution must cap numeric score at `650` and cap limit treatment to starter schedule.
- Household-business scoring without owner consent is forbidden.

## Out of Scope

- EBM challenger model execution in the production lending path
- Any dynamic or generative explanation authoring beyond approved-code translation
- Consumer-lending-specific policies unrelated to SME / household merchant underwriting
- Cross-border data transfer design beyond the consent/retention requirements in this spec

## Dependencies

- Verified connectors for bank, e-wallet, POS, marketplace, and self-reported feeds
- Entity-resolution service with merge support and duplicate detection
- Versioned scorecard package containing bins, WoE tables, coefficients, and validation report
- Audit log storage with immutable write semantics
- Consent registry for PDPD owner-level permissions
